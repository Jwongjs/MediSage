SP1 — Q1: Where should the LLM run in production?

The local Llama 3.1-8B GGUF must go. What replaces it?

A

Managed Cloud API

Claude (Anthropic) · OpenAI GPT-4o · Gemini

Call an API per request. No GPU, no server management. Pay per token.

Pros

  • Zero infra to manage
  • Instant horizontal scale
  • Best model quality
  • Simple Docker image (no CUDA)

Cons

  • PHI leaves your server — BAA required
  • Per-token cost at scale
  • Latency depends on provider
B

Private GPU VPS / Serverless Inference

RunPod · Modal · Together.ai · Replicate

Your own GPU endpoint running an open-source model. Data stays on your infra.

Pros

  • PHI stays on your infra
  • No BAA needed with provider
  • Cost-effective at volume
  • Full model control

Cons

  • GPU rental cost (~$0.20–0.50/hr)
  • Cold-start latency on serverless
  • More infra to manage
  • Model quality vs GPT-4o gap
C

Hybrid: Cloud API + Local EfficientNet

Claude/OpenAI for LLM · EfficientNet stays containerized

Cloud API handles all text reasoning. EfficientNet skin classifier stays in a sidecar container (only ~50MB, no GGUF).

Pros

  • Best reasoning quality
  • EfficientNet is lightweight (~50MB)
  • Realistic portfolio architecture
  • Simple Docker image for LLM service

Cons

  • Text PHI still goes to API provider (BAA needed)
  • Two inference paths to maintain

My recommendation: C (Hybrid) — cloud API for reasoning quality + EfficientNet local since it's tiny and already trained. BAA with Anthropic/OpenAI is free for developer accounts.
